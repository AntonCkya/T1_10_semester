# Part 1. Декомпозиция, модули, интеграции, HLD

## DDD-анализ

### Домен

Система работает в области **радиовещания и вовлечения аудитории**. Основной агрегат — **эфир** (broadcast): диджей ведёт передачу, играет треки, слушатели взаимодействуют в реальном времени.

### Поддомены и bounded contexts

Используем Event Storming для выявления ключевых событий и агрегатов:

| Поддомен | Тип | Ключевые события | Агрегаты |
|----------|-----|-------------------|----------|
| **Управление эфиром** | Core | TrackChanged, BroadcastStarted, BroadcastEnded | Broadcast, OnAirTrack |
| **Голосование** | Core | VoteCasted, VoteAggregated, PlaylistVoted | Vote, VotingSession |
| **Заказы песен** | Core | SongRequested, RequestAccepted, RequestRejected, RequestPlayed | SongRequest |
| **Каталог треков** | Supporting | TrackAdded, TrackUpdated, TrackSearched | Track |
| **Идентификация** | Generic | UserRegistered, UserLoggedIn, RoleAssigned | User, Session |
| **Уведомления** | Generic | NotificationSent, PushDelivered | Notification |
| **Аналитика** | Supporting | ReportGenerated, MetricsCollected | Report, Metric |
| **Чат** | Supporting | MessageSent, MessageModerated | ChatMessage |

## Сервисы (с обоснованием)

### 1. **Сервис авторизации и аутентификации (Auth Service)**
- **Обоснование**: Единая точка входа для управления пользователями (слушатели, диджеи, админы). Разные роли требуют разных прав доступа. Требование ИБ — централизованный контроль доступа (2FA для диджеев/админов).
- **Ответственность**: Регистрация, логин, JWT-токены, управление ролями (RBAC), 2FA.
- **Технология**: Keycloak (готовое решение, поддерживает OAuth 2.0, RBAC, 2FA из коробки).

### 2. **Сервис эфира (Broadcast Service)**
- **Обоснование**: Core-домен. Отвечает за синхронизацию с системой вещания (ICEcast/AzuraCast) и отслеживание текущего состояния эфира. Выделяем отдельно, так как интеграция с внешней системой вещания — самостоятельная зона ответственности с собственными SLA.
- **Ответственность**: Получение метаданных текущего трека, история эфира, расписание эфиров, управление состоянием (active/inactive).
- **Интеграция**: Polling/Webhook от ICEcast → публикация событий `TrackChanged` в Kafka.

### 3. **Сервис голосования (Voting Service)**
- **Обоснование**: Core-домен с высокими требованиями к производительности (до 150 RPS в пик) и real-time агрегации. Выделен отдельно от эфира, так как имеет иную модель данных (голоса vs треки) и иную нагрузочную характеристику (write-heavy).
- **Ответственность**: Приём лайков/дизлайков за текущий трек, голосование за плейлист, агрегация результатов в реальном времени, защита от накруток (rate limiting, дедупликация).
- **Особенность**: Наиболее нагруженный сервис, требует горизонтального масштабирования.

### 4. **Сервис заказов (Request Service)**
- **Обоснование**: Отдельный бизнес-процесс с собственной моделью данных (статусная машина: ожидает → принят → отклонён/исполнен) и лимитами (не более 3 заказов/час на пользователя). Выделен отдельно от голосования, так как заказ — это запрос с состоянием, а голос — идемпотентное действие.
- **Ответственность**: Приём заказов, валидация (лимиты, дубли, фильтр контента), маршрутизация к активному диджею, управление статусами.

### 5. **Сервис каталога (Catalog Service)**
- **Обоснование**: Поддерживающий домен. Каталог треков — справочник с полнотекстовым поиском. Выделяем отдельно, так как поиск по каталогу использует специализированную СУБД (Elasticsearch) и имеет собственную модель данных, независимую от эфира.
- **Ответственность**: Хранение метаданных треков, полнотекстовый поиск, автодополнение, синхронизация с внешними базами (MusicBrainz).

### 6. **Сервис уведомлений (Notification Service)**
- **Обоснование**: Cross-cutting concern. Реализован как отдельный сервис для унификации каналов доставки (WebSocket, Push) и развязки производителей событий от потребителей.
- **Ответственность**: Управление WebSocket-соединениями, рассылка push-уведомлений (FCM/Huawei), подписки пользователей на события (любимый диджей вышел в эфир, заказ принят).

### 7. **Сервис аналитики (Analytics Service)**
- **Обоснование**: Поддерживающий домен. Агрегация данных из других сервисов для дашбордов администраторов и диджеев. Выделяем отдельно, так как использует OLAP-подход (ClickHouse) и не должен нагружать транзакционные БД.
- **Ответственность**: Сбор бизнес-метрик, построение дашбордов, рейтинг диджеев, топ запрашиваемых треков.

### 8. **Сервис чата (Chat Service)** — MVP- / опционально
- **Обоснование**: Социальная функция, не критична для MVP. Выделяем отдельно из-за высокой частоты сообщений и необходимости модерации.
- **Ответственность**: Приём/доставка сообщений, модерация (автофильтр), история чата.

## Интеграции между сервисами

### Способы взаимодействия

| # | Сервис-источник | Сервис-приёмник | Тип | Протокол | Обоснование |
|---|-----------------|-----------------|-----|----------|-------------|
| 1 | Auth Service | Все сервисы | Синхронная | REST (JWT validation) | Каждый запрос требует проверки токена. JWT stateless — верификация без обращения к Auth Service (публичный ключ) |
| 2 | Broadcast Service | Notification Service | Асинхронная | Kafka (event: `TrackChanged`) | Смена трека — событие, которое нужно доставить всем подключённым клиентам. Kafka обеспечивает надёжность и буферизацию |
| 3 | Broadcast Service | Voting Service | Асинхронная | Kafka (event: `TrackChanged`) | Голосование привязано к текущему треку. При смене трека Voting Service сбрасывает счётчики и создаёт новую сессию голосования |
| 4 | Voting Service | Notification Service | Асинхронная | Kafka (event: `VoteAggregated`) | Агрегированные результаты голосования пушатся всем WebSocket-клиентам через Notification Service |
| 5 | Request Service | Notification Service | Асинхронная | Kafka (event: `RequestStatusChanged`) | Статус заказа меняется — уведомление слушателю (push) и диджею (WebSocket) |
| 6 | Catalog Service | Request Service | Синхронная | REST (gRPC internal) | При заказе песни нужно валидировать наличие трека в каталоге. Синхронный вызов, так как результат нужен immediately |
| 7 | Frontend (слушатели) | Voting Service | Синхронная | REST + WebSocket | REST для отправки голоса, WebSocket для получения real-time обновлений |
| 8 | Frontend (слушатели) | Request Service | Синхронная | REST | CRUD операции с заказами |
| 9 | Frontend (диджеи) | Request Service | Синхронная | REST + WebSocket | REST для управления статусами заказов, WebSocket для получения новых запросов в real-time |
| 10 | Frontend (диджеи) | Broadcast Service | Синхронная | REST | Управление эфиром (начать/закончить, опубликовать плейлист) |
| 11 | Все сервисы | Analytics Service | Асинхронная | Kafka (events) | Все бизнес-события стекаются в Analytics для OLAP-обработки |
| 12 | Notification Service | Mobile clients | Асинхронная | WebSocket + FCM/Huawei | WebSocket для подключённых клиентов, Push — для отключённых |

### Обоснование выбора протоколов:

- **REST (HTTP/2)**: Для CRUD-операций и запрос-ответных взаимодействий. Простота, хорошая инструментальная поддержка, совместимость с OpenAPI 3.0.
- **WebSocket**: Для real-time доставки данных клиентам (результаты голосования, текущий трек, чат). Minimises latency, avoids polling overhead.
- **Kafka**: Для внутренней асинхронной коммуникации между сервисами. Надёжная доставка, буферизация при пиковой нагрузке (спайк в 50K пользователей), возможность ретроспективного анализа событий (Analytics).
- **gRPC**: Для внутренних синхронных вызовов между сервисами (Catalog ↔ Request), где важна низкая latency и эффективная сериализация (protobuf).

### Описание потока данных (основные сценарии):

**1. Смена трека в эфире:**
```
ICEcast → (webhook) → Broadcast Service → Kafka [track-changed]
    → Notification Service → WebSocket → все подключённые клиенты
    → Voting Service → сброс счётчиков, новая сессия голосования
    → Analytics Service → запись в ClickHouse
```

**2. Голосование слушателя:**
```
PWA → REST → API Gateway → Voting Service → Redis (INCR) → Kafka [vote-events]
    → Notification Service → WebSocket → дашборд диджея + все слушатели
```

**3. Заказ песни:**
```
PWA → REST → API Gateway → Request Service → REST → Catalog Service (валидация)
    → Request Service → Kafka [request-events]
    → Notification Service → WebSocket → панель диджея
    → Push → слушателю (статус обновлён)
```

# Part 2. Выбор баз данных, репликация, шардинг

## Алгоритм выбора БД

Для каждого сервиса определяем:
1. Характер нагрузки (read/write ratio, тип запросов)
2. Требования к консистентности (strong vs eventual)
3. Объём данных и рост
4. Специфические требования (full-text search, time-series, real-time counters)

---

## 1. Auth Service — **PostgreSQL**

| Критерий | Обоснование |
|----------|-------------|
| Характер нагрузки | Read-heavy (проверка токенов через публичный ключ — без запроса к БД), редкие записи (регистрация, смена пароля) |
| Консистентность | Strong consistency — нельзя допустить рассинхронизацию учётных данных |
| Данные | Реляционные: пользователи, роли, разрешения, сессии |
| Объём | < 200 КБ записей (100K MAU) — минимальный |

**Репликация**: Primary + 2 synchronous replicas (для отказоустойчивости).
**Шардинг**: Не требуется — объём данных мал.

---

## 2. Broadcast Service — **PostgreSQL + Redis**

### PostgreSQL (основное хранилище):

| Критерий | Обоснование |
|----------|-------------|
| Характер нагрузки | Write: при смене трека (~12 записей/час). Read: запросы текущего эфира (~80 RPS в пик) |
| Консистентность | Strong — текущее состояние эфира должно быть актуальным |
| Данные | Эфиры, история треков, расписание, привязка диджеев |

### Redis (кэш текущего состояния эфира):

| Критерий | Обоснование |
|----------|-------------|
| Зачем | Текущий трек, активный диджей — "горячие" данные, запрашиваемые при каждом открытии приложения |
| TTL | Обновляется при каждом событии `TrackChanged` (каждые ~5 мин) |
| Паттерн | Cache-aside: приложение сначала идёт в Redis, при miss — в PostgreSQL |

**Репликация PostgreSQL**: Primary + 1 sync replica + 1 async replica.
**Шардинг**: Не требуется — один город, объём данных мал.

---

## 3. Voting Service — **Redis (primary) + ClickHouse (archive)**

### Redis Cluster (real-time счётчики):

| Критерий | Обоснование |
|----------|-------------|
| Характер нагрузки | Write-heavy: до 150 RPS голосов в пик. Read: агрегированные результаты для real-time обновлений |
| Консистентность | Eventual — допустима задержка в доли секунды при агрегации |
| Данные | Счётчики лайков/дизлайков по текущему треку, сессии голосования |
| Паттерн | `INCR` для счётчиков, `SET` с TTL для сессии голосования |

### ClickHouse (архивация и аналитика):

| Критерий | Обоснование |
|----------|-------------|
| Зачем | Голоса — временные ряды. ClickHouse идеально для аналитических запросов ("топ треков за неделю", "динамика голосования") |
| Запись | Батчевая вставка из Kafka → ClickHouse (раз в 10-30 сек) |
| Хранение | Партиционирование по дням, TTL на сырые данные (90 дней), агрегаты хранятся вечно |

**Репликация Redis**: Redis Sentinel (1 master + 2 replicas) для HA + автоматический failover.
**Шардинг Redis**: Не требуется на данном масштабе (один город, ~150 RPS пик). При росте — Redis Cluster с шардированием по `track_id`.

---

## 4. Request Service — **PostgreSQL**

| Критерий | Обоснование |
|----------|-------------|
| Характер нагрузки | Умеренная: ~7 500 записей/день, read — диджей запрашивает очередь (каждые 5 сек × 10 диджеев = 2 RPS) |
| Консистентность | Strong — статусная машина заказа требует транзакционности (ожидает → принят → исполнен) |
| Данные | Реляционные: заказы, статусы, лимиты пользователей, история |
| Запросы | Поиск по user_id (активные заказы), по dj_id (очередь заказов), по статусу |

**Репликация**: Primary + 1 sync replica + 1 async replica.
**Шардинг**: Не требуется. При масштабировании — шардирование по `dj_id` (каждый диджей работает со своим shard).

---

## 5. Catalog Service — **PostgreSQL + TypeSense / Elasticsearch**

### PostgreSQL (основное хранилище):

| Критерий | Обоснование |
|----------|-------------|
| Характер нагрузки | Read-heavy: поиск при каждом заказе, автодополнение при вводе |
| Данные | Реляционные: треки, исполнители, жанры, связи |

### TypeSense / Elasticsearch (полнотекстовый поиск):

| Критерий | Обоснование |
|----------|-------------|
| Зачем | Нечёткий поиск по названию трека и исполнителю, автодополнение, fuzzy matching |
| Объём | ~100K-500K треков — малый, но поиск должен быть быстрым (< 100 мс) |
| Синхронизация | CDC из PostgreSQL (Debezium) → Kafka → TypeSense |

**Репликация PostgreSQL**: Primary + 2 replicas.
**Шардинг**: Не требуется — каталог треков целиком помещается в память.

---

## 6. Notification Service — **Redis + Kafka**

| Критерий | Обоснование |
|----------|-------------|
| Redis | Хранение состояния WebSocket-соединений (user_id → connection mapping), pub/sub для рассылки |
| Kafka | Consumer группы для обработки событий из других сервисов |
| Данные | Не хранит бизнес-данные постоянно — stateless сервис с Redis для routing |

**Репликация Redis**: Redis Sentinel (разделяется с Voting Service или отдельный кластер).
**Шардинг**: Не требуется.

---

## 7. Analytics Service — **ClickHouse**

| Критерий | Обоснование |
|----------|-------------|
| Характер нагрузки | Write-heavy (batch inserts из Kafka), Read — аналитические запросы к дашбордам |
| Консистентность | Eventual — аналитика допускает задержку в несколько секунд |
| Данные | Time-series: голоса, заказы, активность пользователей, прослушивания |
| Запросы | Агрегации (SUM, COUNT, AVG) по временным окнам, GROUP BY, топ-N |

**Репликация**: ReplicatedMergeTree — 2 ноды, каждая хранит полную копию (ClickHouse встроенная репликация через ZooKeeper).
**Шардинг**: Не требуется на данном масштабе. При росте — шардирование по дате (по месяцам).

---

## 8. Chat Service (опционально) — **Redis + PostgreSQL**

### Redis (real-time):
- Pub/Sub для доставки сообщений подключённым клиентам.

### PostgreSQL (история):
- Хранение истории чата для модерации и отображения при переподключении.

# Part 3. Дополнительные компоненты (MUST / SHOULD)

## 1. Обязательные компоненты (MUST)

| Компонент | Обоснование | Реализация |
|-----------|-------------|------------|
| **Load Balancer** | Распределение трафика между инстансами сервисов. Без него горизонтальное масштабирование невозможно.尤其 важно для Voting Service при пиковых нагрузках (50K concurrent) | Yandex Application Load Balancer (ALB) |
| **API Gateway** | Единая точка входа для клиентов. Rate limiting (защита от DDoS и накруток), маршрутизация запросов к нужным сервисам, terminate TLS | Yandex ALB (встроенный) + custom rate-limiting middleware |
| **Кэш (Redis)** | Уменьшение нагрузки на БД для "горячих" данных: текущий трек эфира, счётчики голосования, сессии пользователей. Критичен для Voting Service — без кэша PostgreSQL не выдержит 150 RPS записи | Redis Sentinel (1 master + 2 replicas) |
| **IdP (Keycloak)** | Централизованная аутентификация и авторизация для всех ролей. OAuth 2.0 + OIDC. 2FA для диджеев/админов — требование ИБ | Keycloak (managed или self-hosted в Yandex Cloud) |
| **WAF** | Защита от OWASP Top 10. Система голосования — мишень для атак (накрутки, SQL injection в поиске каталога). Требование сертификации ИБ | Qrator Labs / Yandex Cloud WAF |
| **CI/CD** | Автоматизация развертывания. Без неё увеличивается T2M и риск человеческого фактора на PROD. Критично для rapid iteration (6 мес MVP) | GitLab CI: build → test → security scan → canary deploy → production |
| **Обзервабилити** | Мониторинг 8 сервисов, Kafka, 3 типов БД. Логирование (Loki), метрики (Prometheus), трейсы (Tempo). Без обзервабилити невозможно диагностировать проблемы в real-time системе | OpenTelemetry Collector → Prometheus + Loki + Tempo + Grafana |
| **Резервное копирование** | Гарантия восстановления данных (RPO < 1 мин). Требование ФЗ-152. Голоса и заказы — бизнес-критичные данные | WAL-G для PostgreSQL + S3 snapshots + ClickHouse backups на S3 |
| **Message Broker (Kafka)** | Развязка сервисов, буферизация при пиковых нагрузках (спайк 50K → 150 RPS голосов), гарантия доставки событий. Без Kafka сервисы жёстко связаны и не выдержат пиковую нагрузку | Kafka (3 brokers) в Yandex Cloud |

## 2. Рекомендуемые компоненты (SHOULD)

| Компонент | Обоснование | Реализация |
|-----------|-------------|------------|
| **Anti-fraud / Bot Detection** | Защита от накрутки голосов — критичная бизнес-проблема для радиостанции. Без неё голосование теряет смысл. Device fingerprinting + behavioral analysis | Custom middleware (Go) + Redis rate limiter + CAPTCHA (hCaptcha через прокси) |
| **CDN** | Доставка статики (JS/CSS бандлы, обложки треков, иконки) с edge-узлов. Снижает latency загрузки PWA для мобильных пользователей | Yandex CDN + S3 бакет |
| **Service Mesh** | Упрощение управления трафиком между 8 микросервисами. Retries, timeouts, circuit breaking, mutual TLS. Снижает операционную сложность | Istio / Linkerd (при self-hosted k8s) или Yandex Service Mesh |
| **Feature Flags** | Безопасный rollout фич (чат, голосование за плейлист, реакции). Позволяет включать/выключать функционал без деплоя, что критично для radio — нельзя допустить падение в эфирное время | GitLab Feature Flags / Unleash |
| **Auto-scaling** | Автоматическое масштабирование Voting Service и Notification Service при пиковых нагрузках (популярный эфир). Без него — или переплата за простаивающие ресурсы, или падение при спайке | Kubernetes HPA (based on RPS + WebSocket connections) или Yandex Cloud Auto-scaling Groups |
| **Secrets Manager** | Централизованное хранение секретов (DB passwords, API keys, JWT signing keys). Требование ИБ-сертификации | Yandex Lockbox / HashiCorp Vault |
| **Container Registry** | Хранение Docker-образов всех сервисов. Необходим для CI/CD pipeline | Yandex Container Registry / GitLab Container Registry |

## Резюме архитектурных решений

| Решение | Обоснование |
|---------|-------------|
| Микросервисная архитектура (8 сервисов) | Разные паттерны нагрузки (Voting = write-heavy, Catalog = read-heavy), разные SLA, независимое масштабирование |
| Go для Core-сервисов | Высокая производительность при обработке concurrent запросов, низкое потребление памяти, отличный стандартный HTTP/WebSocket stack |
| Kafka как event backbone | Буферизация при пиках нагрузки, развязка продьюсеров и консьюмеров, возможность ретроспективного анализа |
| Redis для голосования | O(1) для INCR операций, идеально для счётчиков real-time |
| ClickHouse для аналитики | Columnar storage,高效 batch inserts, built-in replication |
| Keycloak для Auth | Готовое решение с OAuth 2.0, RBAC, 2FA — не пишем велосипед |
| Kubernetes / Yandex Managed | Auto-scaling Voting Service при спайках, managed сервисы снижают эксплуатационную нагрузку |
